# 公交线路路段空洞问题：根因分析与修复留档

> 时间：2026-06-08 · 涉及需求2（公交线路加工）+ 需求3（GPS↔路段 SDK）

## 1. 现象

生成 192 路（上行）的路段时，链路 `5130138420598800924`(seq18) 与 `5130138420598801039`(seq19)
之间**缺了一节**，A 终点与 B 起点之间留下约 **68.2 米**的空洞。进一步核查发现：这是**系统性问题**，
不止 192——3 条线路 5 个方向共 **139 个 gap(>20m)**。

## 2. 根因（两类，互相独立）

### 根因一：`_drop_against_track` 用「全局最近点」判方向，去/回程重叠时误删

192 上行去程与回程都走龙州路，两条腿近似反向平行、相距仅 ~20–30m。
[`_drop_against_track`](../amap_service/sdk/conversion.py) 原本拿每个路段的朝向去和**整条轨迹的全局最近点**比：

| 路段 | 段中点最近的轨迹顶点 | 落在哪条腿 | 与轨迹夹角 | 结果 |
|---|---|---|---|---|
| A (seq18) | 顶点25 | 去程 | 2° | 保留 |
| **龙州路桥接** | **顶点251** | **回程** | **179°** | **误删** |
| B (seq19) | 顶点28 | 去程 | 171°(被 doubled 掩码救下) | 保留 |

桥接 link 在**去程**匹配上（GPS 离它仅 0.6m），朝向本是对的，但它几何中点的全局最近顶点恰好落在
**回程腿**（反向），夹角 179° → 被当成「对向车道错配」删掉。本该兜底的 `_doubled_track_mask` 在该顶点
因两腿相距略超 20m 而漏标，保护失效。

### 根因二：matcher 无「连通性保证」，中间短 link 从未被选中

逐 GPS 点贪心选最近/最便宜 link，输出整条 link，但**不保证相邻被选 link 在拓扑上首尾相接**。
192 上行 11 个「可补」空洞里，8 个的桥接 link GPS 离它仅 0.4–5.4m（远在 30m 容差内），却在它覆盖的
每个点上都被旁边平行/更长/更低 roadclass 的 link 比下去 → 永远不成段 → 留洞。代码里没有任何一步去补
这些缺失的连接 link。

> 一句话：当前是「逐点选最近 link」而非「在路网图上求一条连通路径」，所以只要某条中间 link 在每个点都
> 被邻居比下去、或事后被清洗删掉，就留下永久空洞。

## 3. 修复（治本）

### 3.1 新增图上补链 —— [`amap_service/sdk/connector.py`](../amap_service/sdk/connector.py)

`GraphConnector`：把 `road_link_coord` 按共享端点建图，对每对首尾不接（gap > `connect_gap_m`）的相邻
路段，用 Dijkstra 求**最短连通路径**，把缺失 link 按行进方向（含 `reverse_coords`）splice 回去。

- 带绕远保护：路径长度超 `max(gap*detour_factor, gap+detour_slack_m)` 或 link 数超 `max_fill_links` 即放弃（宁留洞不乱补）。
- 查询按缺口 bbox 局部加载，配合 `road_link_coord(longitude,latitude)` 索引，开销小。
- **同时解决根因一、二**：无论 link 是「没被选中」还是「被清洗删了」，补链都会把它接回来。

### 3.2 修 `_drop_against_track` —— 弧长窗口比对局部轨迹

改为按**弧长进度**只比对该路段对应的**局部轨迹段**（`against_window_frac` / `against_window_m`）。
去/回程虽在空间上重叠，但在弧长上相距很远，加窗后前向段只与去程腿比 → 不再误删。

### 3.3 接入与配置

- [`conversion.py`](../amap_service/sdk/conversion.py)：`linetrack_to_segments` 清洗后追加 `connector.fill(...)`；`linetrack_to_linkinfos` 同样走补链。
- [`config/schema.py`](../amap_service/config/schema.py) 新增 4 个可调参数，透传至 [`transit_build`](../amap_service/pipelines/transit_build.py) 管线，并写入 `config/config.yaml(.example)`：

| 参数 | 默认 | 含义 |
|---|---|---|
| `connect_gap_m` | 8 | 相邻路段端点距超此值即视为缺口 → 触发补链 |
| `max_fill_links` | 8 | 单个缺口最多补入路段数（防绕远） |
| `against_window_frac` | 0.2 | 反向判定只比对局部轨迹：占总长比例 |
| `against_window_m` | 80 | ...且至少此米数 |

- [`db/schema.py`](../amap_service/db/schema.py)：新增 `idx_road_link_coord_lnglat(longitude,latitude)` 空间索引（SDK 候选匹配 + 补链 bbox 查询提速）。
  注意：`create_all` 不会给**已存在**的表补索引，旧库需手动建一次：
  `CREATE INDEX IF NOT EXISTS idx_road_link_coord_lnglat ON road_link_coord(longitude,latitude);`

## 4. 验证

真实数据 gap(>20m) 数：

| 线路/方向 | 修复前 | 修复后 |
|---|---|---|
| 192/up | 24 | 7 |
| 47/up | 24 | 4 |
| 47/down | 25 | 8 |
| 89/up | 40 | 11 |
| 89/down | 26 | 7 |
| **合计** | **139** | **37** |

补回 102 个。剩余 37 个中 **36 个路网里根本不存在可连接两端的 link**（真实转弯拐角/数据断裂，理论下限），
仅 1 个属连接器边角情形。192 龙州路那一节已正确补回。

测试：新增 [`tests/sdk/test_connector.py`](../tests/sdk/test_connector.py)（6 例）+ [`test_conversion.py`](../tests/sdk/test_conversion.py)
3 例（含把 192 误删场景缩成最小复现的窗口回归用例）。全量 **155 项测试通过**。
