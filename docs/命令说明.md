# 命令说明（测试运行手册）

本服务的全部命令都通过 `python -m amap_service`（或安装后的 `amap-service`）调用。
两种写法等价，本文统一用 `python -m amap_service`，方便在源码目录直接测试。

```bash
python -m amap_service <子命令> [-c 配置文件]
amap-service        <子命令> [-c 配置文件]   # pip install -e . 之后可用
```

- `-c / --config`：配置文件路径，默认 `config/config.yaml`。
- 子命令一览：`initdb`、`run-once <job>`、`run`、`serve`、`match-report`。
- `run-once` 的 `<job>` 取值：`road-network`、`traffic`、`transit`、`transit-build`、`section-build`。

---

## 0. 准备工作（首次）

```bash
# 1) 安装依赖（开发模式，含 dev/可选项）
pip install -e ".[dev]"          # 仅核心
pip install -e ".[dev,mysql]"    # 若用 MySQL

# 2) 准备配置文件
cp config/config.yaml.example config/config.yaml
# 敏感项可用环境变量覆盖（双下划线分层），例如：
export AMAP__TRANSIT__PASSWORD=真实密码

# 3) 建表（创建 road_link / road_link_coord / traffic_status / transit_segment 等）
python -m amap_service initdb
```

> 提示：`run-once` 和 `run` 启动时都会自动 `init_db` 建表，所以单独跑 `initdb` 不是必须的，但首次想确认表结构时可以单独执行。

## 1. 初始化路网（全量路网落地，需求1）

从 `areaLinkPub` 接口拉取全量路网，解析 `coordList` 写入 `road_link` + `road_link_coord`，按 `link_id` upsert。

```bash
python -m amap_service run-once road-network
```

- 上游：`GET {amap.endpoint}/g5_server/map/api/areaLinkPub`
- 解析模式：`parse_mode: stream`（响应大，流式解析省内存）
- 是**线路路段转换（transit-build）的前置条件**：SDK 做空间匹配时依赖 `road_link_coord` 已填充。

---

## 2. 获取实时路况（全量实时路况，需求1）

从 `traffic/status` 接口拉取实时路况，按 `link_id` upsert 到 `traffic_status` 并刷新 `updated_at`。

```bash
python -m amap_service run-once traffic
```

- 上游：`GET {amap.endpoint}/g5_server/map/api/traffic/status`
- 解析模式：`parse_mode: memory`（响应小，一次性解析）
- 若 `redis.enabled: true`，会按 `redis.uses` 启用最新路况快照 / 增量变更检测（跳过未变 link）。

> 提速与实时供数：`traffic` 入库走批量 upsert（约 40 万行秒级）。开启 `redis.uses.incremental_detection`
> 只写变化的 link、`latest_traffic_snapshot` 写热缓存；第三方按 link 查实时路况用 SDK 的
> `TrafficReader.get_latest_traffic(link_ids)`（Redis 命中→DB 回落，返回最后一次抓取值）。
> 注意：以上 Redis 快速路径须先在配置中设置 `redis.enabled: true`（示例配置默认为 false，需手动开启并提供 Redis 服务；未开启时自动回落直连 DB）。

---

## 3. 公交线路抓取与路段转换（需求2）

公交是两阶段流水线，建议先跑 `transit` 看清真实响应结构，再跑 `transit-build` 落库。

### 3a. 阶段一：抓取并存档原始响应

```bash
python -m amap_service run-once transit
```

- 链路：`GetToken`（签名鉴权）→ `GetLineFilterNow`（线路列表）→ `GetRoadLineEntity`（逐条线路对象）。
- 作用：抓取并存档原始响应，便于确认 `token_path` / `line_name_path` / `line_name_field` 等 dot-path 配置是否正确。

### 3b. 阶段二：初始化线路路段转换（GPS 轨迹 → 有序路段）

```bash
python -m amap_service run-once transit-build
```

- 对每个方向的 `LineLonLat` 轨迹调用 SDK 的 `TrackConverter.linetrack_to_linkinfos`，得到有序 `LinkInfo`。
- 按 `(line_name, direction)` 整段替换写入 `transit_segment`；同时把线路实体的站级数据写入 `transit_station`。
- **依赖**：必须先成功跑过 [第 1 步「初始化路网」](#1-初始化路网全量路网落地需求1)，否则 SDK 没有 `road_link_coord` 可匹配。
- 单条线路失败会跳过并继续其余线路，不中断整次运行。
- **末尾自动触发 [`section-build`](#section-build站间路段占比)**：转换完成后立即重算站间路段占比，所以单跑这一条命令即可产出 `transit_segment` + `transit_station` + `transit_section_link` 全部数据。
- 线路筛选由 `config.yaml` 的 `transit.companys` / `transit.lines` / `transit.line_limit` 控制（探查时可把 `line_limit` 设为 5 等小值）。

## 4. 启动长驻调度（cron daemon，需求1）

按配置中的 cron 表达式定时执行各 job（默认：路网每天 01:00、路况每 2 分钟、公交每天 03:00）。

```bash
python -m amap_service run
```

- 计划完全由配置驱动，不硬编码；可在 `config.yaml` 的各 job `cron` / `enabled` 字段调整。
- 前台运行，`Ctrl+C` 退出。

## 5. 本地 Mock 服务器（无需连真实上游即可测试需求1）

`demo/mock_api_server.py` 从抓包日志 `demo/amap_api_log.txt` 提取两个 GET 接口的真实出参，本地起服务返回。

```bash
# 起 mock（默认监听 0.0.0.0:8080）
python3 demo/mock_api_server.py
python3 demo/mock_api_server.py --port 9000     # 换端口
python3 demo/mock_api_server.py --rebuild       # 强制重新从日志提取缓存
python3 demo/mock_api_server.py --extract-only   # 只提取缓存不起服务
```

配合主服务：把 `config.yaml` 的 `amap.endpoint` 指向 mock，再跑路网 / 路况：

```yaml
amap:
  endpoint: "http://127.0.0.1:8080"
```

```bash
python3 demo/mock_api_server.py &        # 后台起 mock
python -m amap_service run-once road-network
python -m amap_service run-once traffic
```

> 公交（需求2）的三个接口在不同主机，mock 服务器未覆盖，需连真实上游或自行 mock。

## section-build（站间路段占比）

从已落库的 transit_segment + transit_station 计算每条线路每个方向的「站间路段长度占比」，
写入 transit_section_link（静态，不含路况）。transit-build 末尾会自动跑一次；也可单独执行：

```bash
python -m amap_service run-once section-build
```

查询层 SDK（StationTrafficResolver，入参线路名称）据此 + 实时 traffic_status，
给前端返回 [{link_id, state, pct}]（pct 为该区间内长度占比，和=100；state 缺失默认 1=畅通）。

## 6. HTTP API 服务（需求3/4/5）

`amap-service serve` 以独立进程启动 HTTP API，需先在配置中设置 `api.enabled: true`。

```bash
python -m amap_service serve
python -m amap_service serve -c config/config.yaml   # 指定配置文件
```

- 监听地址由 `api.host` 和 `api.port` 控制（默认 `0.0.0.0:8080`）。
- OpenAPI 交互文档在 `/docs`（Swagger UI）。
- 鉴权：`api.auth.enabled: true` 时，业务接口需在请求头携带 `X-API-Key`（值由 `api.auth.api_key` 或环境变量 `AMAP__API__AUTH__API_KEY` 设置）；健康检查 `/health` 始终不鉴权。

### API 端点（前缀 `/api/v1`）

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/health` | 健康检查（不鉴权） |
| `GET` | `/lines` | 已建线路清单 |
| `GET` | `/lines/{line_name}/segments?direction=` | 需求3 线路静态路段列表 |
| `GET` | `/lines/{line_name}/traffic?direction=&geometry=false` | 需求4 线路逐路段实时路况 |
| `GET` | `/lines/{line_name}/sections?direction=` | 需求5 整条线路站间占比+路况 |
| `GET` | `/lines/{line_name}/sections/{to_level_id}?direction=` | 需求5 单个站间区间详情 |

**通用约定：**

- `direction` 参数省略=返回全部方向，传 `0` 或 `1` 过滤；传非法值返回 422。
- `geometry=true` 时响应中才带 `line_track` 坐标字段（默认 `false`，瘦 payload）。
- 线路未建（无 `transit_segment` 记录）返回 404。
- 鉴权开启时，业务接口缺少或错误的 `X-API-Key` 返回 401。
- `link_id` 字段一律以字符串形式返回（防 JavaScript 64 位整数精度丢失）。

## 7. MQTT 发布（需求4/5）

`mqtt.enabled: true` 时，`amap-service run`（cron daemon）在每轮路况刷新完成后，自动全量发布以下两个主题（`prefix` 由 `mqtt.topic_prefix` 配置，默认 `amap`）：

| 主题 | 说明 |
|------|------|
| `{prefix}/line/{line_name}/traffic` | 需求4 线路逐路段实时路况 |
| `{prefix}/line/{line_name}/section` | 需求5 线路站间占比+路况 |

- `mqtt.enabled: false`（默认）时，路况定时任务行为不变，不产生任何 MQTT 连接。
- `qos`、`retain`、`include_geometry` 均由配置控制；`include_geometry: false` 为瘦 payload（不含 `line_track`）。
- `publish_map`（需求4）和 `publish_section`（需求5）可独立开关，默认均为 `true`。
- 连接超时由 `connect_timeout_seconds` 控制（默认 5 秒）；连接失败不影响数据库写入流程。

## 8. 线路匹配长度报表（match-report）

统计每条**已匹配线路**各方向的「原始 GPS 轨迹总长」与「匹配成路段后的总长」对比，输出 CSV，便于核查匹配质量（差异过大通常意味着该方向匹配有偏差）。

```bash
python -m amap_service match-report
python -m amap_service match-report -o logs/line_match_report.csv   # 指定输出路径
```

- 默认输出 `logs/line_match_report.csv`（可用 `-o / --output` 覆盖）；编码 UTF-8-SIG，Excel 直接识别中文。
- 数据来源（只读，无需 Redis / 上游）：
  - **原始轨迹长度**：`transit_line_raw` 中每条线**最新一条**响应的 `LineLonLat`（各方向）折线总长。
  - **匹配后长度**：`transit_segment` 中该方向所有 seg 的 `line_track`（按行进方向裁剪后的路段）折线长之和。
- CSV 列：

  | 列 | 说明 |
  |------|------|
  | 线路名称 | 线路号，如 `47` |
  | 上下行 | `上行`（direction=0，含单环）/ `下行`（direction=1） |
  | 原始轨迹长度 | 原始 GPS 轨迹总长（米，1 位小数） |
  | 匹配后长度 | 匹配成路段后的总长（米，1 位小数） |
  | 差异百分比 | `(匹配后 - 原始) / 原始 × 100`（2 位小数）。通常为小幅负值（匹配后略短于含抖动的原始轨迹） |

- 每条已匹配 `(线路, 方向)` 一行；若该方向缺原始轨迹（`transit_line_raw` 无对应响应），原始轨迹长度与差异百分比两列留空。
- **前置**：需先跑过 [`transit-build`](#3b-阶段二初始化线路路段转换gps-轨迹--有序路段)（产出 `transit_segment`），且 `transit_line_raw` 有归档响应（由 [`transit`](#3a-阶段一抓取并存档原始响应) 或 `transit-build` 写入）。
