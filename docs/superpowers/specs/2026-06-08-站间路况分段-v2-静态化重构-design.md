# 站间路况分段 v2：静态化重构设计

> 时间：2026-06-08 · 归属：需求2（公交线路加工）+ 需求3（SDK）
> 前置：v1 已实现 `StationTrafficResolver`（线路对象进，实时跑几何 + 叠路况）。
> 见 v1 设计：`docs/superpowers/specs/2026-06-08-站间路况分段-design.md`

## 1. 动机

线路站级是相对静态的数据：只要线路站级与匹配出的路段链不变，**站间路段的组成与长度占比就不变**。而路况 `state` 每 2 分钟实时变化。v1 把两者耦合在一次查询里（每个请求都重跑链拼接 + 站点单调对齐 DP + 切区间），不合理。

**v2 把"确定性的昂贵计算"物化到表，把"易变的路况"在读取时叠加。**

收益：
- 查询从"每请求跑几何算法"变为"查表 + 一次路况 join"，性能差一个数量级。
- 昂贵的对齐每次 build 跑一遍，不是每请求跑。
- 查询入参从"线路对象"变为"线路名称"，解耦上游线路实体结构。
- 静态数据未来可直接进可选 Redis 缓存（线名为 key、重建时失效），路况单独取。

## 2. 分层数据流

```
road-network ──▶ transit-build ──▶ section-build ──▶ [查询层 SDK]
 (路网落库)      拉线路实体:        纯 DB→DB:          线名进, 叠路况
                · transit_segment   读 segment+station   出 {link_id,state,pct}
                · transit_station    跑几何算法
                                     写 transit_section_link
```

- **transit-build**（已有，扩展）：除写 `transit_segment`，再把线路实体的 `Stations` 落到 `transit_station`。
- **section-build**（新）：纯 DB→DB，读 `transit_segment`（链）+ `transit_station`（站坐标），跑几何算法，写 `transit_section_link`。**不碰路况**。可独立 `run-once section-build` 重跑；也在 transit-build 末尾自动跑一次。
- **查询层 SDK**：线名进，查静态表 + join 路况。不跑任何几何。

## 3. 新增数据表

### 3.1 `transit_station`（站级静态数据）

| 字段 | 类型 | 说明 |
|---|---|---|
| id | INTEGER PK | |
| line_name | TEXT NOT NULL | 线路号 |
| nor_code | TEXT | 规范码 NorCode |
| direction | INTEGER NOT NULL | 0 上行/单环, 1 下行 |
| level_id | INTEGER NOT NULL | 站级号 |
| level_name | TEXT | 站名 |
| longitude | REAL NOT NULL | 站点经度（Lon02） |
| latitude | REAL NOT NULL | 站点纬度（Lat02） |
| created_at | TIMESTAMP | |

UNIQUE(line_name, direction, level_id) · INDEX(line_name, direction)

### 3.2 `transit_section_link`（站间路段占比）

| 字段 | 类型 | 说明 |
|---|---|---|
| id | INTEGER PK | |
| line_name | TEXT NOT NULL | 线路号 |
| nor_code | TEXT | 规范码 |
| direction | INTEGER NOT NULL | 0/1 |
| from_level_id | INTEGER NOT NULL | 区间起点站级 |
| to_level_id | INTEGER NOT NULL | 区间终点站级（**方法一入参就是它**） |
| seq | INTEGER NOT NULL | 该 link 在本区间内的顺序（0 起） |
| link_id | BIGINT NOT NULL | 路段 ID（64 位） |
| length_m | REAL NOT NULL | 该 link 在本区间内的长度（米，事实源） |
| pct | INTEGER NOT NULL | 预计算占比（本区间内和=100） |
| built_at | TIMESTAMP | 构建时间 |

UNIQUE(line_name, direction, to_level_id, seq) · INDEX(line_name, direction)

设计要点：
- **同时存 `length_m` 与 `pct`**：length_m 是事实源（可重算/聚合/调试），pct 是渲染直接用的预计算值。
- **存 `from_level_id` + `to_level_id` 两个**，不假设 `to-1`：当前"上一站"是 LevelId 排序后的前一个，不一定等于 `level_id-1`（LevelId 理论上可能不连续）。
- **保留 `seq`、link_id 不设唯一**：回头路时同一 link_id 会在不同区间、甚至同一区间出现两次，seq 排序且允许重复。
- **路况绝不进此表**：state 永远查询时叠加。

## 4. 文件结构与职责

| 文件 | 改动 |
|---|---|
| `db/schema.py` | + `transit_station`、`transit_section_link` 两张表 |
| `db/repositories.py` | + `replace_transit_stations(engine, line_name, direction, nor_code, stations)`、`replace_transit_section_links(engine, line_name, direction, nor_code, rows)`，均按 (line_name, direction) 事务整体替换，复用 `replace_transit_segments` 模式 |
| `parsing/transit.py` | + `parse_line_stations(raw)`：从 GetRoadLineEntity 响应按方向取 `[{line_name, nor_code, direction, stations:[(level_id, level_name, lng, lat)]}]`，类比 `parse_line_tracks` |
| `sdk/section_compute.py`（新） | 纯几何：从 `station_traffic.py` 迁入 `ChainLink`/`build_chain`/`sample_chain`/`align_stations`/`section_links`/`largest_remainder`，+ 新函数 `compute_section_rows(chain, stations)` → `[(from_level_id, to_level_id, seq, link_id, length_m, pct), ...]`（**无路况**）。`_data_of`/`_line_name_of`/`_stations_for` 不再需要（站点来自 DB），删除 |
| `sdk/station_traffic.py` | 瘦身为**读取器** `StationTrafficResolver`：查 `transit_section_link` + `traffic_status` |
| `pipelines/section_build.py`（新） | `run_section_build(engine, config) -> stats`：遍历 `transit_segment` 里出现的 (line, direction)，载 segment + station，建链，`compute_section_rows`，整体替换写入 |
| `pipelines/transit_build.py` | 解析并写 `transit_station`；处理完所有线路后调用 `run_section_build` |
| `cli.py` / `scheduler/runner.py` | 注册 `section-build` 子命令（与 road-network / transit-build 同形） |

## 5. 查询层 SDK 契约（入参改线名，返回不变）

```python
StationTrafficResolver(engine, default_state: int = 1)   # 不再需要 sample_step_m

def station_section(self, line_name: str, direction: int, level_id: int) -> list[dict]:
    """查 transit_section_link WHERE line_name, direction, to_level_id=level_id ORDER BY seq
    → 批量取 traffic_status → 叠 state（缺省 default_state）。
    -> [{"link_id", "state", "pct"}, ...]，pct 之和=100；无行返回 []。"""

def line_sections(self, line_name: str) -> dict:
    """一次查全线 section_link（ORDER BY direction, to_level_id, seq）+ 一次查全部 link 路况 → 分组。
    -> { direction: [ {to_level_id: [ {link_id, state, pct}, ... ]}, ... ] }；无行返回 {}。"""
```

- 查询层**不跑几何**：纯查表 + 一次路况 `IN` 批量 join。
- `state` 缺失默认 `default_state`（1）。
- pct 直接取表中预计算值（构建时已用最大余额法保证和=100）。

## 6. 构建器算法（section-build 内）

复用 v1 既有纯函数（迁至 `section_compute.py`）：

1. `build_chain(segments)`：把该方向 `transit_segment`（按 seq）拼成带累计弧长的链。
2. `sample_chain` + `align_stations`：把 `transit_station` 该方向站点（按 level_id 升序）单调对齐到链上，得每站弧长位置（解决回头路/对开/环线消歧）。
3. 对每对相邻站 `(prev, cur)`：`section_links(chain, arc_prev, arc_cur)` 得 `[(link_id, length_m), ...]`；`largest_remainder` 算 pct；产出带 `from_level_id=prev.level_id`、`to_level_id=cur.level_id`、`seq` 的行。
4. `replace_transit_section_links` 按 (line, direction) 整体写入。

`compute_section_rows(chain, stations)` 封装步骤 2~3，`stations` 为 `[(level_id, lng, lat), ...]`（已按 level_id 升序），返回行元组列表。**纯函数、无 DB、无路况**。

## 7. 迁移

新表由 `init_db`（`create_all`）自动创建（只建不存在的表，旧库安全）。落地一次：

```bash
python -m amap_service run-once transit-build   # 填 transit_segment + transit_station，末尾自动 section-build
# 或单独：
python -m amap_service run-once section-build    # 仅重算 transit_section_link（需 segment+station 已就绪）
```

## 9. 不做（YAGNI）

- 静态表不存几何坐标（line_track）：比例色条只要 pct + state。要画地图实际折线时再加。
- 不在静态表存路况。
- 暂不接 Redis 缓存（架构已为其留好位置，启用是后续独立工作）。
- 不为静态表做输入指纹/增量构建：每次随 transit-build 整体重建（输入同源，避免漂移）；将来要增量时再加 `built_at` 之上的指纹判断。
